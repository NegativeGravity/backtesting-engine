# Candle-by-Candle Execution Semantics

## Unit of advancement

One manual `step_forward` advances exactly one synchronized close batch. With `M1` as the execution timeframe, every batch contains the newly closed M1 bar and may also contain M5, M15, H1, H4, or D1 bars that close at the same timestamp.

No subsequent batch is read until the current batch, broker processing, strategy callbacks, actions, feedback, chart output, and account snapshot are complete.

## Exact event order

For every synchronized close batch, the engine performs this sequence:

1. The data engine emits only bars whose close time is the current event time.
2. The broker processes the current execution-timeframe bar.
3. Existing market, limit, stop, stop-loss, and take-profit orders are evaluated.
4. Fills, position changes, realized PnL, floating PnL, margin, and account state are updated.
5. Broker events generated by the execution bar are delivered to the strategy.
6. The strategy receives the closed bars in deterministic subscription order.
7. The strategy may emit order actions, position-management actions, chart commands, and structured logs.
8. Strategy actions are validated and executed by the broker.
9. Broker feedback from those actions is delivered back to the strategy in bounded feedback rounds.
10. The engine publishes one live replay frame.
11. The engine waits for the next manual step or playback clock.

This ordering makes signals based on a closed candle execute according to the configured policy. The default policy is `next_bar_open`; a market order emitted after candle N closes can first fill on the open of execution candle N+1.

## No look-ahead

A strategy can access only:

- warmup bars strictly before the run start;
- closed subscribed bars at or before the current event time;
- forming higher-timeframe bars only when `forming_allowed` is explicitly configured;
- broker state resulting from data already processed.

Future bars never enter the worker request. Higher-timeframe bars configured as `closed_only` are invisible until their official close time.

## Play, pause, and speed

`play` repeatedly invokes the same one-batch step operation. Speed changes only the delay between completed steps; it never changes event ordering or execution semantics.

## Step backward and seek

Candle data and strategy state are not mutated backward in place. Rewind is deterministic:

1. stop the current isolated strategy process;
2. create a new broker and strategy process from the original run inputs;
3. replay synchronized batches from the beginning;
4. stop exactly at the requested batch or progress point;
5. publish a fresh bootstrap snapshot.

This is slower than an approximate in-memory rollback but guarantees that indicators, strategy variables, orders, positions, drawings, and account state match a clean run to the same candle.

## Intrabar boundary

The platform is candle-accurate, not tick-path accurate. When a candle touches multiple executable levels and their true ordering cannot be known, the configured intrabar ambiguity policy is applied and the ambiguity is recorded.


## Strategy source isolation

Before `on_start`, the engine snapshots the selected external strategy package into the run working directory. All worker starts, including deterministic rewind, prepend this snapshot to the worker import path. Finalization reruns the strategy from the same snapshot and rejects the run if the live and finalized deterministic digests differ.
