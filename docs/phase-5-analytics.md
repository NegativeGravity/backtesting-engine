# Phase 5 — Analytics

Phase 5 adds a deterministic analytics engine and a dedicated result workspace to the replay dashboard.

## Data sources

Analytics are calculated from two authoritative sources inside each replay bundle:

- completed `Trade` entities
- execution-bar equity points written by the broker observer

The replay timeline keeps sparse account snapshots for fast state reconstruction. The analytics equity table stores one row per execution bar so drawdown and return calculations do not depend on replay snapshot frequency.

## Generated artifact

Each replay run contains:

```text
data/replay/runs/<run_id>/analytics-report.json
```

The report is also stored in SQLite metadata and referenced by `ReplayBundleManifest.analytics_report_path`.

## Metrics

### Performance

- gross profit and gross loss
- gross and net P&L
- total return
- CAGR for runs with at least the configured minimum duration
- profit factor
- expectancy
- average and median trade
- average win and loss
- payoff ratio
- best and worst trade
- average and median R-multiple
- System Quality Number when enough R observations exist

### Risk

- maximum drawdown amount and percentage
- drawdown duration and recovery duration
- average drawdown
- daily annualized volatility
- Sharpe and Sortino ratios
- Calmar ratio
- recovery factor
- ulcer index
- historical VaR and CVaR at the configured confidence level
- best and worst daily return

### Trade behavior

- win, loss and breakeven counts
- long and short counts
- win and loss rates
- average, median and maximum holding time
- maximum consecutive wins and losses
- merged time-in-market exposure
- ambiguous intrabar trade count
- average MAE and MFE
- MFE capture for profitable trades

### Execution costs

- commission
- spread cost
- slippage cost
- swap
- total execution cost
- average cost per trade
- cost as a percentage of gross profit
- cost-component shares

## Periodic analytics

Daily returns use the final execution-bar equity point of each UTC day. Monthly and yearly rows include opening equity, closing equity, return, realized trade P&L, trade counts, win rate and local maximum drawdown.

UTC is used deliberately because the engine stores canonical UTC timestamps. Broker-session analytics can be added later as a separate timezone-aware view without changing the canonical report.

## Breakdowns

Trade breakdowns are generated by:

- side
- symbol
- exit reason
- UTC weekday
- UTC exit hour

Each row contains trade count, win rate, net P&L, average P&L, profit factor, average R and average holding time.

## Distributions

Deterministic histograms are produced for:

- net trade P&L
- realized R-multiple
- holding duration

## Point-in-time analytics

The API supports full-run and replay-cursor scopes:

```text
GET /api/runs/{run_id}/analytics
GET /api/runs/{run_id}/analytics?end_time_ns=<cursor>
```

Cursor reports include only equity points and completed trades available at or before the requested time. The dashboard refreshes cursor analytics after replay movement is paused.

## Run comparison

```text
GET /api/analytics/compare?run_id=<run_1>&run_id=<run_2>
```

The comparison contract exposes net P&L, return, drawdown, Sharpe, Sortino, profit factor, win rate and trade count.

## Dashboard

The top workspace switch exposes Replay and Analytics modes. Analytics mode contains:

- headline KPI cards
- exact equity and drawdown charts
- monthly performance heatmap
- performance, risk, behavior and cost panels
- side, exit-reason, weekday and hourly breakdown tables
- P&L, R and duration distributions
- drawdown episode table
- full-run and replay-cursor scopes

## Numerical policies

- calculations are deterministic for identical inputs
- monetary values remain `Decimal` at contract boundaries
- statistical formulas use daily percentage returns
- Sharpe uses sample standard deviation and 252-day annualization by default
- Sortino uses downside deviation relative to the configured daily risk-free rate
- CAGR is suppressed for runs shorter than the configured minimum duration
- VaR and CVaR use the historical empirical tail
- missing or insufficient samples produce `null`, not fabricated zero-quality ratios

## Fidelity boundary

Analytics describe the broker simulation output. They cannot improve execution fidelity beyond the candle-level assumptions of the broker simulator. A precise metric over imprecise fills is still limited by the underlying execution model.
